# Coding Convention
- 코딩 컨벤션은 [캠퍼스 핵데이 Java 코딩 컨벤션](https://naver.github.io/hackday-conventions-java/)을 따릅니다. (css, html, js는 제외)
- InteliJ에서 코딩 컨벤션을 적용할때는 [codeconvention 디렉토리](/codeconvention) 내 xml 파일을 사용해주세요.

# Commit 규칙
- 커밋 컨벤션은  [Conventional Commits](https://www.conventionalcommits.org/ko/v1.0.0/) 규칙을 따릅니다.
- 아래 내용을 참고하세요.
### 🚀 기능 개발 및 버그 수정

- **`feat` (Feature):** **새로운 기능을 추가**했을 때 사용합니다.
    - *예시:* `feat: 로그인 기능 구현`
- **`fix` (Bug Fix):** **버그나 오류를 수정**했을 때 사용합니다.
    - *예시:* `fix: 프로필 이미지 업로드 실패 오류 수정`

### 🛠️ 코드 리팩토링 및 성능 개선

- **`refactor`:** 기능의 변화(추가/수정) 없이 **코드를 개선**했을 때 사용합니다. 가독성을 높이거나, 중복 코드를 제거하거나, 구조를 변경했을 때가 해당합니다.
    - *예시:* `refactor: 중복되는 인증 로직 함수화`
- **`perf` (Performance):** 사용자가 느끼는 기능은 그대로지만, **성능을 향상**시켰을 때 사용합니다. (속도 개선, 메모리 절약 등)
    - *예시:* `perf: 메인 페이지 이미지 로딩 속도 최적화`

### 🎨 코드 포맷팅 및 테스트

- **`style`:** 코드 의미에 영향을 주지 않는 **스타일 변경** 시 사용합니다. (포맷팅, 세미콜론 누락 수정, 공백 제거, 변수명 변경 등 - CSS 디자인 수정이 아닙니다!)
    - *예시:* `style: 린터 규칙에 맞게 세미콜론 추가 및 들여쓰기 수정`
- **`test`:** **테스트 코드를 추가하거나 수정**했을 때 사용합니다. 프로덕션(실제 서비스) 코드에는 영향을 주지 않는 작업입니다.
    - *예시:* `test: 회원가입 성공 케이스 유닛 테스트 추가`

### 📦 빌드, CI/CD 및 내부 시스템 관련

- **`build`:** **빌드 시스템이나 외부 의존성(라이브러리)을 수정**했을 때 사용합니다. (Webpack, Maven, Gradle, npm 패키지 추가/삭제 등)
    - *예시:* `build: lodash 라이브러리 추가`
- **`ci`:** **CI/CD 설정 파일을 수정**했을 때 사용합니다. (GitHub Actions, Jenkins, Travis CI 설정 등)
    - *예시:* `ci: 배포 파이프라인에 테스트 단계 추가`
- **`chore`:** 위의 카테고리에 속하지 않는 **기타 자질구레한 작업**이나 유지보수 작업에 사용합니다. (.gitignore 수정, 주석 제거, 패키지 설정 변경 등)
    - *예시:* `.gitignore에 로그 파일 경로 추가`

### 📝 문서 및 변경 취소

- **`docs` (Documentation):** **문서를 수정**했을 때 사용합니다. (README.md, 주석, 위키 등)
    - *예시:* `docs: README.md에 설치 방법 최신화`
- **`revert`:** 이전에 했던 커밋을 취소(되돌리기)하고 싶을 때 사용합니다. Git revert 명령어를 쓰면 자동으로 이 태그가 붙기도 합니다.
    - *예시:* `revert: "feat: 로그인 기능 구현" 커밋 되돌림`

### 💡 중요한 접미사 규칙: `!` (Breaking Change)

태그 종류는 아니지만, Conventional Commits에서 매우 중요한 표준 규칙 중 하나는 바로 느낌표(`!`)입니다.

어떤 태그든 뒤에 `!`를 붙이면 "이 커밋은 기존 시스템과 호환되지 않는 중대한 변경사항(Breaking Change)이 있다"는 것을 뜻합니다. 주로 API 구조가 통째로 바뀌거나 유저 데이터 구조가 바뀔 때 사용하며, 시맨틱 버저닝(Semantic Versioning)에서 **Major 버전**을 올리는 기준이 됩니다.

    - *예시:* `feat!: 기존 로그인 API 제거 및 OAuth 2.0 전면 도입`
    - *예시:* `refactor!: 구버전 데이터베이스 스키마 마이그레이션 및 필드 삭제`

# Pull Request 규칙
- PR을 올리기 전 PR 템플릿 내용을 따라 한 번 더 점검 후 형식에 맞게 작성해주세요.  

## 제목
별도의 태그 없이 작업물에 대해서 한 줄로 요약해 주세요. 맨 뒤에 이슈 번호를 붙입니다.
## What
- 이 필드에는 자신이 어떤 작업을 했는지 정리합니다
## Why
- 이 필드에는 이 작업이 왜 필요한지에 대해 정리합니다.

## Checklist
### 필수 사항
- [] CodeStyle 플러그인으로 캠퍼스 핵데이 Java 코딩 컨벤션에 위배되지 않는지 확인하셨나요?
- [] Test가 통과되었는지 확인하셨나요?
- [] 빌드가 되는지 확인했나요?
- [] 민감한 데이터가 들어가 있는지 확인하셨나요?
### 기능 관련
- [] 변경되거나 새로 작성된 기능에 관한 테스트 코드를 작성하거나 수정하셨나요?
- [] 변경되거나 새로 작성된 기능에 관한 문서를 작성하거나 수정하셨나요?
### 기타 사항
- [] 새로운 의존성을 추가하거나 수정하셨나요?
- [] 중대한 변경사항(Breaking Change)이 있나요?
